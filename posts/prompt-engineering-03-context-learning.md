# 提示词工程（三）：上下文学习策略

## 前言

大语言模型最神奇的能力之一，就是能够从上下文中学习，无需额外的参数更新，仅通过几个示例就能理解任务并完成类似的任务。这种能力被称为**上下文学习（In-Context Learning）**。

本文将深入探讨上下文学习的三种核心模式：
- **Zero-shot Learning**：零样本学习
- **One-shot Learning**：单样本学习
- **Few-shot Learning**：少样本学习

你将学习到如何设计高质量的示例、如何排列示例顺序、以及如何在有限的上下文窗口内最大化学习效果。

## 一、上下文学习概述

### 1.1 什么是上下文学习

**上下文学习（In-Context Learning, ICL）** 是指大语言模型在不更新参数的情况下，仅通过 Prompt 中提供的示例或指令，就能理解新任务并做出相应的能力。

### 1.2 为什么上下文学习有效

```
┌─────────────────────────────────────────────────────────────┐
│                    预训练阶段                                 │
│  模型在海量文本中学习语言模式、知识和推理能力                    │
│                     ↓                                        │
│              获得通用能力                                      │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    推理阶段                                   │
│  通过 Prompt 中的示例/指令激活相关能力                         │
│                     ↓                                        │
│  模型理解任务模式 → 应用到新输入                               │
└─────────────────────────────────────────────────────────────┘
```

核心原理：
1. **模式识别**：模型从示例中提取输入-输出的映射模式
2. **知识激活**：示例激活了预训练阶段学到的相关知识
3. **任务推断**：模型推断出要执行的任务类型

### 1.3 三种学习模式对比

| 模式 | 示例数量 | 适用场景 | 优势 | 局限 |
|------|----------|----------|------|------|
| Zero-shot | 0 | 通用任务、简单分类 | 简洁、Token 消耗低 | 对复杂任务效果有限 |
| One-shot | 1 | 格式明确的任务 | 快速展示输出格式 | 可能过拟合到单个示例 |
| Few-shot | 2-10 | 复杂任务、需要特定风格 | 展示多种模式、提高准确性 | Token 消耗高 |

## 二、Zero-shot Learning

### 2.1 什么是 Zero-shot

**Zero-shot Learning** 指不给出任何示例，仅通过指令让模型完成任务。

### 2.2 Zero-shot 适用场景

- 任务类型是模型预训练时已熟悉的
- 输出格式可以通过文字描述清楚
- 任务相对简单，不需要复杂的模式学习

### 2.3 Zero-shot 实例

**情感分析**
```
请判断以下评论的情感倾向（积极/消极/中性）：

评论：这款手机的拍照效果令人惊艳，夜景模式特别强大！
```

**文本分类**
```
请将以下新闻分类到对应的类别：科技/财经/体育/娱乐/其他

新闻：苹果发布最新款 MacBook Pro，搭载 M3 芯片，性能提升 40%。
```

**翻译任务**
```
请将以下英文翻译成中文：

The quick brown fox jumps over the lazy dog.
```

### 2.4 Zero-shot 提示技巧

**使用明确的指令词**
```
✅ 明确的指令
"请将以下文本翻译成中文："
"请判断这段评论的情感倾向："
"请总结这段文字的核心观点："

❌ 模糊的指令
"看看这段话"
"这个怎么办"
```

**提供清晰的输出格式描述**
```
请对以下文章进行分类。

可选类别：
- 技术
- 商业
- 生活方式
- 其他

输出格式：
类别：[选择最符合的类别]
置信度：高/中/低
理由：简要说明分类依据
```

## 三、One-shot Learning

### 3.1 什么是 One-shot

**One-shot Learning** 指在 Prompt 中提供一个示例，展示输入和期望的输出格式。

### 3.2 One-shot 适用场景

- 需要展示特定的输出格式
- 任务有一定特殊性，但模式相对简单
- Token 预算有限

### 3.3 One-shot 实例

**实体提取**
```
请从文本中提取人名和地名。

示例：
输入：马云在杭州创建了阿里巴巴公司。
输出：
- 人名：马云
- 地名：杭州

输入：李彦宏毕业于北京大学，后来创立了百度公司。
输出：
```

**文本改写**
```
请将以下文本改写成更正式的商务用语。

示例：
原文：你们的报价太高了，我们接受不了。
改写：经过审慎评估，贵司的报价超出了我方预算范围，希望能进一步协商。

原文：这个方案不行，我们需要重新考虑。
改写：
```

### 3.4 One-shot 设计技巧

**选择具有代表性的示例**
- 覆盖典型的输入模式
- 展示清晰的输出格式
- 避免边缘情况作为唯一示例

**示例与任务的相关性**
```
✅ 好的示例
示例和实际任务属于同一类型，只是内容不同

❌ 差的示例
示例是情感分析，实际任务却是文本分类
```

## 四、Few-shot Learning

### 4.1 什么是 Few-shot

**Few-shot Learning** 指在 Prompt 中提供多个示例（通常是 2-10 个），让模型从多个示例中学习任务模式。

### 4.2 Few-shot 核心优势

1. **模式泛化**：从多个示例中提取通用模式
2. **边界理解**：通过不同示例理解任务边界
3. **风格学习**：学习特定的输出风格或格式
4. **复杂推理**：支持需要多步推理的任务

### 4.3 Few-shot 基础示例

**情感分析**
```
请判断以下评论的情感倾向。

示例 1：
评论：这家餐厅的服务态度太差了，等了一个小时才上菜！
情感：消极

示例 2：
评论：产品质量一般，和描述有些出入。
情感：消极

示例 3：
评论：非常满意，物流很快，包装也很精美。
情感：积极

示例 4：
评论：这款手机的性能中规中矩，没有什么特别突出的地方。
情感：中性

评论：客服态度很好，耐心解答了我的所有问题。
情感：
```

### 4.4 Few-shot 设计原则

#### 原则 1：示例多样性

提供不同类型的示例，帮助模型理解任务边界。

```
任务：判断用户意图（查询/购买/投诉/其他）

示例 1（查询）：
输入：这款手机支持 5G 吗？
意图：查询

示例 2（购买）：
输入：我要买两台，什么时候能发货？
意图：购买

示例 3（投诉）：
输入：收到的商品有破损，要求退货退款！
意图：投诉

示例 4（其他）：
输入：谢谢你的帮助。
意图：其他
```

#### 原则 2：输入-输出一致性

所有示例的输入和输出格式必须一致。

```
✅ 一致的格式
示例 1：
输入：...
输出：...

示例 2：
输入：...
输出：...

❌ 不一致的格式
示例 1：
Q: ...
A: ...

示例 2：
输入：...
输出格式：[...]
```

#### 原则 3：难度递进

将简单示例放在前面，复杂示例放在后面。

```
示例 1：简单情况（基准示例）
示例 2：简单情况（确认模式）
示例 3：复杂情况（扩展能力）
示例 4：边界情况（定义边界）
```

#### 原则 4：平衡正负样本

对于分类任务，保持正负样本的平衡。

```
任务：判断是否为垃圾邮件

✅ 平衡的示例
示例 1：垃圾邮件
示例 2：正常邮件
示例 3：垃圾邮件
示例 4：正常邮件

❌ 不平衡的示例
示例 1：垃圾邮件
示例 2：垃圾邮件
示例 3：垃圾邮件
示例 4：正常邮件
```

## 五、示例工程（Example Engineering）

### 5.1 示例选择策略

#### 策略 1：基于相似度的选择

选择与当前输入最相似的示例。

```
实现步骤：
1. 计算当前输入与候选示例的相似度（如余弦相似度）
2. 选择相似度最高的 K 个示例
3. 将这 K 个示例加入 Prompt
```

#### 策略 2：基于多样性的选择

选择覆盖不同模式的示例。

```
实现步骤：
1. 对示例进行聚类
2. 从每个聚类中选择代表性示例
3. 确保示例覆盖不同的场景和边界情况
```

#### 策略 3：困难示例挖掘

选择模型容易出错的示例。

```
实现步骤：
1. 在验证集上测试模型表现
2. 记录模型预测错误的示例
3. 将这些困难示例加入 Prompt
```

### 5.2 示例排序技巧

示例的顺序会影响模型输出！

#### 研究发现

1. **近因效应（Recency Bias）**：模型倾向于模仿最后几个示例
2. **常见类别偏见**：如果某类示例过多，模型可能偏向该类别
3. **顺序敏感性**：不同的示例顺序可能导致不同的输出

#### 排序策略

```
策略 1：简单到复杂
示例 1：基础情况
示例 2：标准情况
示例 3：复杂情况
示例 4：边界情况

策略 2：高频到低频
示例 1：最常见的类型
示例 2：次常见的类型
...
示例 N：稀有类型

策略 3：相关度排序
示例 1：与当前输入最相关
示例 2：与当前输入次相关
...
```

### 5.3 示例标签设计

#### 标签的清晰度

```
✅ 清晰的标签
情感：积极 / 消极 / 中性
类别：技术 / 商业 / 生活
意图：查询 / 购买 / 投诉

❌ 模糊的标签
结果：好 / 不好
类型：A / B / C
```

#### 标签的解释性

```
为标签添加简要说明，帮助模型理解。

示例：
评论：这家店太棒了！
情感：积极（明显正面词汇："棒"）

评论：一般般吧。
情感：中性（无明显情感倾向）
```

## 六、高级技巧：动态示例检索

### 6.1 静态 vs 动态 Few-shot

**静态 Few-shot**：Prompt 中使用固定的示例
```
无论输入什么，都使用同样的 3 个示例
```

**动态 Few-shot**：根据输入动态选择示例
```
输入 A → 选择示例 1、2、3
输入 B → 选择示例 4、5、6
```

### 6.2 实现动态示例检索

```python
from sklearn.metrics.pairwise import cosine_similarity
import numpy as np

class DynamicFewShot:
    def __init__(self, examples, embeddings):
        """
        examples: 候选示例列表
        embeddings: 示例的向量表示
        """
        self.examples = examples
        self.embeddings = embeddings
    
    def retrieve(self, query_embedding, k=3):
        """检索最相似的 k 个示例"""
        similarities = cosine_similarity(
            [query_embedding], 
            self.embeddings
        )[0]
        
        # 获取相似度最高的 k 个索引
        top_k_indices = np.argsort(similarities)[-k:][::-1]
        
        return [self.examples[i] for i in top_k_indices]
    
    def build_prompt(self, query, query_embedding):
        """构建带有动态示例的 Prompt"""
        relevant_examples = self.retrieve(query_embedding)
        
        prompt = "请完成以下任务。\n\n"
        for i, example in enumerate(relevant_examples, 1):
            prompt += f"示例 {i}:\n"
            prompt += f"输入：{example['input']}\n"
            prompt += f"输出：{example['output']}\n\n"
        
        prompt += f"输入：{query}\n输出："
        return prompt
```

### 6.3 向量检索的实际应用

```
系统架构：

用户输入 → 向量化 → 相似度计算 → 选择 Top-K 示例 → 构建 Prompt → 调用 LLM
    ↑                                                    ↓
    └──────────── 示例库（预计算向量） ←──────────────────┘
```

## 七、上下文窗口管理

### 7.1 窗口限制的挑战

```
问题：Few-shot 示例太多，超出上下文窗口限制

上下文窗口 = System Prompt + Few-shot 示例 + 用户输入 + 输出预留
```

### 7.2 解决方案

#### 方案 1：示例压缩

```
原始示例：
输入：这家餐厅的食物非常美味，服务员态度也很好，环境干净整洁，
      价格合理，总的来说是一次很棒的用餐体验，下次还会再来。
输出：积极

压缩后：
输入：餐厅食物美味，服务好，环境佳，价格合理，用餐体验很棒。
输出：积极
```

#### 方案 2：分层检索

```
第一层：粗粒度筛选（关键词匹配）
第二层：细粒度排序（语义相似度）
第三层：选择 Top-K 加入 Prompt
```

#### 方案 3：示例摘要

```
对于长示例，可以只保留关键部分。

原文：一篇 500 字的产品评价
摘要：保留核心观点和情感倾向的关键句子（50 字）
```

## 九、常见陷阱与解决方案

### 陷阱 1：示例偏见（Example Bias）

```
问题：模型过度模仿示例中的特定词汇或模式

示例：
示例 1：这个产品的优点是...缺点是...
示例 2：这款应用的优点是...缺点是...

输入：请评价这部电影。
输出：这部电影的优点是...缺点是...
（即使评价电影用"优缺点"不太合适）

解决方案：
1. 提供多样化的表达方式
2. 使用抽象的示例而非具体的
3. 在指令中强调灵活性
```

### 陷阱 2：标签不平衡

```
问题：某类标签的示例过多，导致模型偏向该类别

示例：
3 个积极示例 + 1 个消极示例

输入：评价中性
输出：积极（被示例影响）

解决方案：
1. 保持标签平衡
2. 明确说明各类别应该是等概率的
```

### 陷阱 3：过度拟合

```
问题：模型机械地模仿示例，缺乏灵活性

示例：
示例的输出都以"综上所述"结尾

模型输出也总是以"综上所述"结尾

解决方案：
1. 示例多样化，避免固定模式
2. 强调"不要机械模仿，要理解任务本质"
```

## 总结

本文深入探讨了上下文学习的核心策略：

1. **Zero-shot Learning**：适用于简单任务，通过清晰的指令让模型直接执行
2. **One-shot Learning**：通过单个示例快速展示任务模式，适合格式明确的任务
3. **Few-shot Learning**：通过多个示例让模型学习任务模式，是处理复杂任务的首选

**关键技巧**：
- 示例多样性：覆盖不同场景和边界情况
- 输入-输出一致性：保持格式统一
- 难度递进：从简单到复杂排列
- 示例排序：利用近因效应优化顺序
- 动态检索：根据输入动态选择最相关的示例

在下一篇文章中，我们将探讨更高级的推理技术——**链式思考（Chain-of-Thought）**，让模型像人类一样一步步思考复杂问题。
